Liv Lakay — preventing payment bypass (production)
==================================================

Goal: a real access code exists in Firestore ONLY after Stripe confirms payment (or you manually mint via Admin SDK).

WHAT BLOCKS CHEATING TODAY (when configured)
---------------------------------------------
  1. codesPurchased documents are created ONLY by:
       • stripeWebhook Cloud Function (after checkout.session.completed + paid)
       • mintLivDevAccessCode (disabled when LIV_DEV_MINT_KEY is empty)

  2. Firestore rules (firestore.rules):
       • allow create, delete, list: if false  on codesPurchased
       • allow get: only if used != true (validate before signup)
       • allow update: only authenticated user marking used=true for themselves
         (affectedKeys limited to used / usedAt / redeemedByUid / redeemedByEmail)

  3. Signup (liv.html):
       • Validates code exists in codesPurchased (or legacy accessCodes)
       • Redeem transaction sets used=true atomically

  4. revealLivAccessCode:
       • Retrieves Stripe session server-side
       • Returns code only if payment_status === "paid"

DEV / TEST BYPASSES (must NOT work in production)
-------------------------------------------------
  These are for local development only:

  A) livPass.html “Buy access code (test)”
       • Enabled only on localhost / 127.0.0.1
       • ?devBypass=1 does NOT enable test mode on keplersiguineau.com

  B) sessionStorage “LIV…” test codes
       • Accepted at signup only when isLivDevBypassUiEnabled() (localhost)

  C) Code TEST at signup
       • Accepted only on localhost / 127.0.0.1
       • LIV_ALLOW_BYPASS_TEST_CODE = false (repo — no production flag path)

  D) mintLivDevAccessCode
       • Leave LIV_DEV_MINT_KEY unset (empty) on liv-lakay production project
       • Function returns 503 dev_mint_not_configured
       • Client LIV_DEV_MINT_KEY = "" — browser never sends a working key

KILL-TEST CHECKLIST (after deploy)
----------------------------------
  1) Live: open /livPass.html?devBypass=1 — NO “(test)” / free mint UI
  2) Signup with a random garbage code — fails (invalid / permission)
  3) Signup with TEST — fails on live (only works on localhost)
  4) One real paid Checkout (or Stripe test mode with real webhook) —
        codesPurchased/{code} appears; reveal returns that code
  5) Sign up once with that code — works; same code a second time — fails (used)
  6) Optional: POST mintLivDevAccessCode with no key — expect 503
        {"error":"dev_mint_not_configured"}

WHY CLIENT-SIDE CAN NEVER BE TRUSTED
------------------------------------
  Anyone can edit JavaScript in the browser. Security lives in:
    • Stripe (payment truth)
    • Cloud Functions + Admin SDK (mint codes)
    • Firestore rules (no client create, single-use redeem)

  Never add a “generate code” button that writes to Firestore from the browser in production.
